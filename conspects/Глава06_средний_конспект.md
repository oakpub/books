# Глава 6 — Средний конспект: Члены и типы в C# / CLR

> **Для кого**: для тех, кто умеет программировать, но не знает C# или .NET.  
> **Что это**: книга Джеффри Рихтера «CLR via C#» — про внутренности платформы .NET на уровне рантайма (CLR = Common Language Runtime).

---

## 1. Члены типа — что может жить внутри класса?

В .NET любой тип (класс, структура) может содержать следующие виды членов. Компилятор превращает их все в **метаданные** (стандартное описание) плюс **IL-код** (промежуточный байткод). Этот формат единый для всех языков платформы — именно поэтому C#, F# и VB.NET могут свободно использовать код друг друга.

### Полный список членов:

| Член | Аналог | Особенности |
|---|---|---|
| **Константа** | `const` / `#define` | Всегда статическая. Значение встраивается при компиляции. |
| **Поле** | instance variable | Может быть `static` (одно на класс) или экземплярным (одно на объект). `readonly` — нельзя менять после конструктора. |
| **Конструктор экземпляра** | `__init__` / ctor | Запускается при `new`. Инициализирует поля объекта. |
| **Конструктор типа** | static initializer | Запускается один раз при первом обращении к типу. Инициализирует статические поля. |
| **Метод** | function / method | Статический (на тип) или экземплярный (на объект). |
| **Перегруженный оператор** | `__add__` / `operator+` | Не входит в CLS (не все языки поддерживают). |
| **Оператор преобразования** | explicit/implicit cast | Не входит в CLS. |
| **Свойство** | getter/setter | Выглядит как поле при использовании, но внутри — методы get/set. Позволяет контролировать доступ к данным. |
| **Событие** | observer / listener | Механизм подписки/отписки на уведомления. Под капотом: поле-делегат + два метода (add/remove). |
| **Вложенный тип** | inner class | Тип внутри типа. Используется для разбиения сложного типа на части. |

## 2. Видимость типа

Когда вы определяете тип на уровне файла (не вложенный):

```csharp
public class OpenForAll { }       // доступен из любой сборки
internal class OnlyInMyLib { }    // доступен только внутри своей dll/exe
class AlsoInternal { }            // по умолчанию тоже internal
```

**Сборка** (.NET assembly) — это скомпилированная библиотека (`.dll`) или приложение (`.exe`). Аналог пакета/модуля в других языках.

---

## 3. Дружественные сборки (`InternalsVisibleTo`)

**Проблема**: Группа A делает библиотеку, группа B её использует. A хочет скрыть внутренние типы от всех, кроме B. Если сделать типы `public` — они откроются вообще всем.

**Решение**: атрибут `InternalsVisibleTo` в сборке A:

```csharp
// В AssemblyInfo.cs сборки A:
[assembly: InternalsVisibleTo("AssemblyB, PublicKey=abc123...")]
```

Теперь сборка B видит `internal`-типы сборки A как `public`. Все остальные — нет.

> **Типичный use-case**: unit-тесты в отдельной сборке, которым нужен доступ к внутренностям тестируемой библиотеки.

## 4. Модификаторы доступа к членам

CLR определяет шесть уровней доступа. В C# реализованы пять:

| C# | CLR-термин | Кто видит |
|---|---|---|
| `private` | Private | Только методы самого типа (и вложенных в него типов) |
| `protected` | Family | Класс + все его наследники (в любой сборке) |
| *(нет)* | Family and Assembly | Наследники только в той же сборке |
| `internal` | Assembly | Любой код в той же сборке |
| `protected internal` | Assembly or Family | Наследники из любой сборки **ИЛИ** любой код в той же сборке |
| `public` | Public | Все и везде |

**Важный момент**: если тип `internal`, то его `public`-методы всё равно недоступны снаружи сборки — потому что снаружи невидим сам тип.

**По умолчанию** в C# члены получают `private`. CLR проверяет доступ не только при компиляции, но и при JIT-компиляции (при запуске). Нарушение вызовет `FieldAccessException` или `MethodAccessException`.

**Правило про наследников**: переопределяя метод базового класса, нельзя сделать его *более закрытым* (нельзя `protected` → `private`), но можно сделать *более открытым* (можно `protected` → `public`). C# запрещает и то, и другое (требует одинаковый модификатор), но CLR это разрешает.

## 5. Статические классы

В C# есть ключевое слово `static` для класса — запрещает создание экземпляров:

```csharp
public static class MathHelper {
    public static double Square(double x) => x * x;
}
```

**Что запрещает компилятор**:
- создание объекта (`new MathHelper()` — ошибка)
- наследование (`static`-класс неявно `abstract` + `sealed`)
- нестатические члены
- использование как тип переменной/параметра

**Когда использовать**: коллекция утилитных функций (как `Math`, `Console`, `File`).

## 6. Частичные типы (`partial`)

```csharp
// Файл 1: Employee.Core.cs
public partial class Employee {
    public string Name { get; set; }
    public void Hire() { ... }
}

// Файл 2: Employee.Validation.cs
public partial class Employee {
    public bool IsValid() { ... }
}
```

Компилятор объединяет эти файлы в **один тип**. CLR вообще не знает о `partial` — она видит единый скомпилированный тип.

**Зачем нужно**:
1. **Совместная разработка**: два программиста редактируют разные файлы одного класса без конфликтов слияния.
2. **Разделение ответственности**: одна часть — бизнес-логика, другая — сериализация и т.д.
3. **Генерация кода**: Visual Studio генерирует UI-код в один файл, программист пишет логику в другой. Случайное изменение сгенерированного кода не ломает дизайнер.

## 7. Компоненты, версионирование и полиморфизм

### Что такое «компонент» в .NET?

Компонент = сборка (`.dll`/`.exe`) со своим версионным номером `major.minor.build.revision`:
- `1.2.3.4` → major=1, minor=2, build=3, revision=4
- Служебная версия (патч) меняет только build/revision: `1.2.1.34`
- Новая несовместимая версия меняет major/minor: `2.0.0.0`

> CLR по умолчанию загружает **точную** версию сборки, указанную при компиляции. Перенаправление возможно, но требует явной настройки.

### Проблема версионирования с виртуальными методами

Представьте:
- **CompanyA** выпустила класс `Phone` с методом `Dial()`
- **CompanyB** создала `BetterPhone : Phone` и добавила свой `Dial()`
- Через год CompanyA добавила в `Phone` метод `EstablishConnection()` — и CompanyB обнаруживает, что её `BetterPhone` тоже имеет `EstablishConnection()`

Возникает конфликт: два одноимённых метода с потенциально разной семантикой.

C# решает это ключевыми словами:

| Ключевое слово | На типе | На методе |
|---|---|---|
| `abstract` | Нельзя создать экземпляр | Нет реализации, наследник обязан переопределить |
| `virtual` | — | Метод можно переопределить в наследнике |
| `override` | — | Метод переопределяет виртуальный метод базового класса |
| `sealed` | Нельзя наследоваться | Нельзя дальше переопределять этот метод |
| `new` | — | Метод скрывает одноимённый метод базового класса (НЕ переопределяет) |

### `override` vs `new` — ключевое отличие

```csharp
class Base {
    public virtual void Hello() => Console.WriteLine("Base");
}

class ChildOverride : Base {
    public override void Hello() => Console.WriteLine("ChildOverride"); // полиморфизм
}

class ChildNew : Base {
    public new void Hello() => Console.WriteLine("ChildNew"); // скрытие
}

Base b1 = new ChildOverride();
b1.Hello(); // → "ChildOverride"  (полиморфизм работает)

Base b2 = new ChildNew();
b2.Hello(); // → "Base"  (new не участвует в полиморфизме!)
```

## 8. Как CLR вызывает методы: `call` vs `callvirt`

В IL-коде (байткоде CLR) есть две инструкции вызова:

| Инструкция | Для чего | Проверяет null? | Полиморфизм? |
|---|---|---|---|
| `call` | статические, экземплярные, виртуальные (явный невиртуальный вызов) | Нет | Нет |
| `callvirt` | экземплярные и виртуальные | **Да** | Для виртуальных — да |

C# почти всегда использует `callvirt` для экземплярных методов — даже невиртуальных. Это нужно для **проверки `null`**: если вызывается метод на `null`-объекте, вы получите `NullReferenceException`, а не неопределённое поведение.

```csharp
Program p = null;
p.GetFive(); // → NullReferenceException в C# (callvirt проверяет null)
```

Исключение: при вызове `base.Method()` компилятор использует `call`, чтобы не попасть в бесконечную рекурсию.

**Значимые типы (struct)** всегда вызываются через `call` — они не могут быть null, и для них нет полиморфизма (они `sealed` на уровне CLR).

## 9. Правила хорошего проектирования (по Рихтеру)

1. **Поля — всегда `private`**. Никаких публичных или защищённых полей. Доступ только через свойства или методы.

2. **Классы — лучше `sealed`** по умолчанию (запечатанные).  
   - Легче версионировать: запечатанный класс можно сделать открытым позже, но не наоборот.  
   - Быстрее: JIT может оптимизировать виртуальные вызовы в невиртуальные.  
   - Безопаснее: наследник не может сломать инварианты базового класса.

3. **Виртуальных методов — как можно меньше**.  
   - Они медленнее невиртуальных.  
   - JIT не может встроить (inline) виртуальные вызовы.  
   - Усложняют версионирование.  
   - Если нужна группа перегрузок — делайте виртуальным только самый «сложный» метод, остальные — невиртуальные обёртки над ним.

4. **Методы и свойства — `private` по умолчанию**, открывать только необходимое.

5. **Вспомогательные типы** лучше делать вложенными (nested), но **не открытыми** (public nested types — антипаттерн).
