# Глава 12 — Средний конспект: Обобщения (Generics)

> **Для кого**: для тех, кто умеет программировать, но не знает C# или .NET.  
> **Что это**: глава книги Джеффри Рихтера «CLR via C#» — про обобщения как механизм CLR, а не просто синтаксис языка.

---

## 1. Что такое обобщения и зачем они нужны

**Обобщения (generics)** — механизм многократного использования **алгоритмов** независимо от типа данных. Это второй (после наследования) ключевой инструмент повторного использования кода в .NET.

Идея: разработчик описывает алгоритм (сортировку, поиск, хранение...) с «заглушкой» вместо конкретного типа. Пользователь подставляет нужный тип.

```csharp
// Автор алгоритма пишет один раз:
public class List<T> {
    public void Add(T item) { ... }
    public T this[int index] { get { ... } }
}

// Пользователь алгоритма указывает конкретный тип:
List<DateTime> dtList = new List<DateTime>();
dtList.Add(DateTime.Now);     // без упаковки
DateTime dt = dtList[0];      // без приведения типа
dtList.Add("строка");         // ошибка компиляции — безопасность типов!
```

### 4 главных преимущества

**1. Безопасность типов.** Компилятор и CLR проверяют, что в `List<DateTime>` нельзя добавить `String`. Ошибка обнаруживается на этапе компиляции.

**2. Защита исходного кода.** Пользователь обобщённого алгоритма не обязан видеть его реализацию (в отличие от шаблонов C++, где нужен `.h`-файл).

**3. Чище и понятнее код.** Нет явных приведений типов, нет лишнего шума.

```csharp
// Без обобщений (ArrayList):
DateTime dt = (DateTime) list[0];  // приведение обязательно

// С обобщениями (List<DateTime>):
DateTime dt = list[0];              // приведение не нужно
```

**4. Производительность.** Без обобщений единственный способ написать «универсальный» алгоритм — использовать `Object`. Для значимых типов это означает **упаковку (boxing)**:

```csharp
ArrayList a = new ArrayList();
a.Add(42);          // упаковка — выделение памяти в куче
int x = (int)a[0]; // распаковка
```

С `List<int>` упаковки нет. Бенчмарк Рихтера: `List<int>` в **7 раз** быстрее `ArrayList of Int32` (1.6 сек vs 10.8 сек на 10 млн операций) и 390 уборок мусора против 6.

## 2. Обобщения в библиотеке FCL

Рихтер рекомендует **всегда** использовать обобщённые коллекции вместо необобщённых (`ArrayList`, `Hashtable` и т.д.). Причины:
- Безопасность типов
- Лучшая объектная модель (меньше виртуальных методов → быстрее)
- Новые возможности только в обобщённых версиях

### Основные пространства имён:

| Пространство имён | Что содержит |
|---|---|
| `System.Collections.Generic` | `List<T>`, `Dictionary<TKey,TValue>`, `SortedDictionary<K,V>`, `Queue<T>`, `Stack<T>`, `HashSet<T>`, `LinkedList<T>`, `SortedSet<T>` |
| `System.Collections.ObjectModel` | `ReadOnlyCollection<T>`, `ObservableCollection<T>` |
| `System.Collections.Concurrent` | `ConcurrentDictionary<K,V>`, `ConcurrentQueue<T>` и др. — для многопоточного кода |

Также класс `System.Array` имеет статические обобщённые методы: `Sort<T>()`, `BinarySearch<T>()`, `Find<T>()`, `ForEach<T>()` и т.д.

## 3. Инфраструктура обобщений в CLR

Обобщения поддерживаются **на уровне CLR** — они не являются просто синтаксическим сахаром компилятора, как в некоторых других языках. Для их реализации Microsoft пришлось изменить IL-команды, формат метаданных, JIT-компилятор, отладчик и другие компоненты платформы.

### Открытые и закрытые типы

- **Открытый тип** (open type) — в нём есть незаданные параметры типа: `List<>`, `Dictionary<,>`. **Нельзя создать экземпляр.**
- **Закрытый тип** (closed type) — все параметры заданы: `List<Int32>`, `List<String>`. **Можно создать экземпляр.**

```csharp
Type t = typeof(Dictionary<,>);           // открытый — нельзя создать
Activator.CreateInstance(t);              // → ArgumentException

Type t2 = typeof(Dictionary<String, Int32>); // закрытый — можно
Activator.CreateInstance(t2);            // OK
```

**Важно:** каждый закрытый тип — это самостоятельный тип со своими статическими полями. `List<Int32>.Count` и `List<String>.Count` — разные поля в разных объектах-типах.

### Разрастание кода (Code Explosion)

При JIT-компиляции CLR генерирует отдельный машинный код для каждой комбинации «метод + тип». Это потенциально раздувает рабочий набор приложения.

CLR применяет две ключевые оптимизации:
1. Один и тот же закрытый тип компилируется **только один раз** — даже если используется в нескольких сборках.
2. Все **ссылочные типы** (классы, интерфейсы, делегаты) используют **один и тот же** машинный код, потому что переменная ссылочного типа — это просто указатель на объект в куче. `List<String>` и `List<Stream>` разделяют один скомпилированный код.
3. Для **значимых типов** — отдельный код на каждый (`List<Int32>` и `List<DateTime>` — разные).

## 4. Обобщённые типы и наследование

Обобщённый тип может наследоваться от других типов как обычно. При закрытии типа аргументом создаётся новый объект-тип с той же иерархией:

- `List<T>` → `Object`, значит `List<String>` → `Object` и `List<DateTime>` → `Object`.

Важное следствие: `List<String>` и `List<DateTime>` — **несовместимые типы**, даже если `String` и `DateTime` оба наследуются от `Object`.

```csharp
// Нельзя:
List<Object> lo = new List<String>(); // ошибка компиляции!

// Можно с вариантностью на интерфейсах:
IEnumerable<Object> eo = new List<String>(); // OK — IEnumerable<T> ковариантен
```

### Псевдонимы для читабельности

```csharp
// Длинно:
System.Collections.Generic.List<System.DateTime> dtList = ...;

// Псевдоним через using (тип не меняется!):
using DateTimeList = System.Collections.Generic.List<System.DateTime>;
DateTimeList dtList = new DateTimeList();

// Или через var:
var dtList = new List<DateTime>();
```

## 5. Обобщённые интерфейсы и делегаты

CLR поддерживает обобщённые интерфейсы и делегаты — это критически важно для работы со значимыми типами без упаковки.

```csharp
// Обобщённый интерфейс:
public interface IEnumerator<T> : IDisposable, IEnumerator {
    T Current { get; }
}

// Обобщённый делегат:
public delegate TResult Func<in T, out TResult>(T arg);
```

### Вариантность: `in` и `out`

Параметры типа обобщённых **делегатов и интерфейсов** могут быть отмечены для задания вариантности:

| Ключевое слово | Термин | Ограничение | Разрешённые преобразования |
|---|---|---|---|
| *(нет)* | Инвариантный | — | Строгое совпадение типов |
| `out T` | Ковариантный | Только на выходе (возвращаемое значение) | Производный → Базовый |
| `in T` | Контравариантный | Только на входе (параметры) | Базовый → Производный |

```csharp
Func<Object, ArgumentException> fn1 = null;

// Ковариантность: ArgumentException → Exception (производный → базовый)
// Контравариантность: Object → String (базовый → производный)
Func<String, Exception> fn2 = fn1; // OK, явного приведения не нужно!
```

> **Ограничение**: вариантность работает **только с ссылочными типами**. `IEnumerable<Object>` не принимает `List<DateTime>`, потому что `DateTime` — значимый тип.

## 6. Обобщённые методы

Методы могут иметь **собственные** параметры типа, независимые от типа-контейнера:

```csharp
// Метод в необобщённом классе:
private static void Swap<T>(ref T o1, ref T o2) {
    T temp = o1;
    o1 = o2;
    o2 = temp;
}

// Метод в обобщённом классе с собственным параметром:
internal sealed class GenericType<T> {
    public TOutput Converter<TOutput>() {
        return (TOutput) Convert.ChangeType(m_value, typeof(TOutput));
    }
}
```

### Выведение типов (Type Inference)

Компилятор C# умеет **автоматически определять** аргументы типа из типов аргументов:

```csharp
int n1 = 1, n2 = 2;
Swap<int>(ref n1, ref n2);  // явное указание типа
Swap(ref n1, ref n2);       // компилятор сам выводит: T = int

string s1 = "A";
object s2 = "B";
Swap(ref s1, ref s2); // ошибка: s1 = String, s2 = Object — тип вывести нельзя
```

При наличии перегрузки **необобщённый метод имеет приоритет**:
```csharp
void Display(String s) { ... }
void Display<T>(T o) { Display(o.ToString()); }

Display("Jeff");          // вызовет Display(String), не Display<T>(T)
Display(123);             // вызовет Display<T>(T), T = Int32
Display<String>("Aidan"); // явно указан обобщённый — вызовет Display<T>(T)
```

## 7. Верификация и ограничения

### Проблема

Без ограничений с переменной типа `T` можно делать только самое базовое: присваивание и вызов методов `Object`:

```csharp
// Не скомпилируется — не все типы имеют CompareTo:
private static T Min<T>(T o1, T o2) {
    if (o1.CompareTo(o2) < 0) return o1; // ошибка CS0117
    return o2;
}
```

### Решение: ограничения (`where`)

```csharp
// Теперь компилятор знает, что T реализует IComparable<T>:
public static T Min<T>(T o1, T o2) where T : IComparable<T> {
    return o1.CompareTo(o2) < 0 ? o1 : o2;
}
```

### Виды ограничений

#### Основные ограничения (не более одного)

| Ограничение | Смысл | Что разрешает |
|---|---|---|
| `where T : class` | T — ссылочный тип | `T x = null;` |
| `where T : struct` | T — значимый тип | `return new T();` (есть конструктор без параметров) |
| `where T : SomeClass` | T — SomeClass или производный | Вызов методов SomeClass |

Запрещено явно указывать `Object`, `Array`, `Delegate`, `ValueType`, `Enum`, `Void`.

#### Дополнительные ограничения (ноль или более)

```csharp
// T должен реализовывать несколько интерфейсов одновременно:
where T : IComparable<T>, IFormattable

// T должен быть совместим с другим параметром типа:
private static List<TBase> ConvertIList<T, TBase>(IList<T> list)
    where T : TBase { ... }
```

#### Ограничение конструктора

```csharp
where T : new()  // T имеет публичный конструктор без параметров

internal sealed class Factory<T> where T : new() {
    public static T Create() => new T(); // OK
}
```

> `struct` и `new()` **не используются вместе** — у всех значимых типов и так есть конструктор без параметров.

### Правила ограничений

- Нельзя перегружать типы/методы **по ограничениям** — только по **арности** (числу параметров типа).
- При переопределении обобщённого виртуального метода ограничения **наследуются** от базового, новые задавать нельзя.
- Нельзя задавать ограничения сразу и `struct`, и `new()` — это избыточно.

## 8. Работа с переменной обобщённого типа: типичные ловушки

### Приведение типа

```csharp
// Нельзя прямо:
private static void Cast1<T>(T obj) {
    Int32 x = (Int32) obj;   // ошибка — T может не быть Int32
    String s = (String) obj; // ошибка
}

// Можно через Object:
private static void Cast2<T>(T obj) {
    Int32 x = (Int32)(Object) obj;   // скомпилируется, но может упасть в рантайме
    String s = obj as String;        // безопаснее для ссылочных типов
}
```

### Присваивание `null`

```csharp
T temp = null;       // ошибка — T может быть значимым типом
T temp = default(T); // OK: null для ссылочных, 0/false для значимых
```

### Сравнение с `null`

```csharp
// Разрешено даже без ограничения:
if (obj == null) { ... }
// JIT просто не сгенерирует код внутри if, если T — значимый тип
```

### Сравнение двух переменных `T`

```csharp
if (o1 == o2) { } // ошибка при неограниченном T — не все типы поддерживают ==
// Решение: добавить ограничение where T : class
```

## 9. Рекомендации по проектированию

1. **Предпочитайте обобщённые коллекции** (`List<T>`, `Dictionary<K,V>`) необобщённым (`ArrayList`, `Hashtable`).
2. Используйте **встроенные делегаты** `Action<...>` и `Func<...>` из FCL вместо определения собственных обобщённых делегатов.
3. При объявлении обобщённого интерфейса или делегата всегда явно указывайте `in`/`out` там, где это возможно — это расширит область применения вашего типа.
4. Имена параметров типа: однобуквенные — `T`, многобуквенные — с префиксом `T` (`TKey`, `TValue`, `TResult`).
